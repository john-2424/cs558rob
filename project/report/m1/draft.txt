Updated Milestone 1 Report Draft
1. Introduction

This project investigates a hybrid robotic manipulation framework that combines classical motion planning with learning-based adaptation for robust task execution. The system uses the Franka Emika Panda 7-DoF manipulator in the PyBullet simulator and is motivated by the complementary strengths of structured planning and machine learning. In the intended final system, sampling-based planning will provide feasible nominal motion, while learning-based residual policies will improve robustness to uncertainty, disturbance, and execution mismatch. The long-term goal is to study whether planner-guided residual reinforcement learning can improve robustness and sample efficiency compared with purely classical or purely learning-based approaches.

Milestone 1 focuses on establishing the classical simulation and execution foundation of the project. The required components for this milestone are: loading the robot system in simulation, constructing the environment, extracting robot and scene state information, and implementing an initial classical motion component. In this work, the classical component is realized as waypoint-based joint-space trajectory execution using interpolated joint targets and PyBullet position control. This provides a concrete motion-execution baseline that can later be extended with configuration-space planning, explicit PD tracking control, and task-level pick-and-place behavior.

The nominal task context for the overall project is tabletop manipulation with a cube object, using a pick-and-place style setup. Accordingly, the first milestone emphasizes a stable and reusable software framework around scene setup, robot state access, joint-space execution, logging, and evaluation, so that future milestones can build on the same architecture without major restructuring.

2. Methods
2.1 Simulation environment setup

The simulation environment was implemented in PyBullet using a fixed-base Franka Emika Panda manipulator. PyBullet was selected because it provides URDF-based robot loading, rigid-body simulation, collision geometry, and a convenient API for both control and later reinforcement learning experiments. The initial scene consists of a ground plane, a rectangular table, a cube manipulation object, and the Panda robot positioned beside the table.

The robot base was positioned at the side of the table and rotated to face the tabletop workspace. The cube was placed on top of the table, producing a realistic tabletop manipulation configuration. This scene was refined iteratively to ensure that the robot, table, and cube were placed in a visually and physically meaningful arrangement before motion execution was added.

2.2 Software organization

The codebase was organized into modular components for simulation, planning, control, trajectories, tasks, learning, perturbations, and evaluation. For Milestone 1, the most important modules were sim/ for environment and robot setup, trajectory/ for joint-space interpolation, evaluations/ for logging and post-processing, and main.py for experiment orchestration. The execution pipeline in main.py loads the environment, corrects object placement, initializes the robot, generates interpolated waypoints, executes forward and return trajectories, and saves trajectory logs for later analysis.

This modular structure was chosen early so that later milestones can add RRT* planning, explicit PD tracking, perturbation injection, and residual reinforcement learning without requiring major code reorganization.

2.3 Robot loading and state extraction

The Panda URDF was loaded with a fixed base, and the simulator was queried to extract joint indices, joint names, joint limits, joint positions, joint velocities, gripper joints, and the end-effector link index. During execution, the system successfully reported seven arm joints and two gripper joints, together with end-effector position and orientation, cube pose, and the table axis-aligned bounding box (AABB). In the latest run, the terminal output confirmed the arm joint indices [0,1,2,3,4,5,6], gripper joint indices [9,10], and end-effector link index 11.

State extraction was performed using PyBullet APIs. End-effector pose was obtained through link-state queries, while cube pose and table geometry were obtained through base-state and AABB queries. This satisfies the milestone requirement of obtaining both robot and environment state information.

2.4 Geometry-based scene placement

AABB-based reasoning was used to improve the consistency of scene placement. After loading the scene, the table’s bounding box was measured and its vertical position was corrected so that the bottom of the table aligned with the ground plane. The cube was then repositioned so that its bottom surface rested on the tabletop height computed from the table AABB rather than using a manually guessed z-value. The implementation in main.py explicitly computes the table bottom height, shifts the table to plane level, computes the tabletop height, and places the cube at that support height with a small clearance.

This geometry-based placement makes the setup more reproducible and reduces sensitivity to mesh-origin offsets and hand-tuned placement values. The latest execution log confirms that the table bottom was at z = 0.0 and the tabletop height was approximately z = 0.052 m.

2.5 Stable robot initialization

After loading, the Panda arm was reset to a nominal home joint configuration and continuously commanded to hold that pose using PyBullet position control. This step was necessary to eliminate the instability observed in earlier scene configurations and to ensure repeatable execution from a known state. In the latest run, the measured joint velocities at initialization were near zero, indicating stable pose holding. The end-effector pose at home was also reported successfully, confirming correct forward-kinematic access.

A stable home configuration is especially important because all trajectory execution in this milestone begins from the same nominal initial pose.

2.6 Joint-space trajectory generation and execution

The main classical component implemented in this milestone is waypoint-based joint-space trajectory execution. A target arm configuration was defined in joint space, and linear interpolation was used to generate a sequence of intermediate waypoints between the home configuration and the target configuration. A matching return trajectory from target to home was generated in the same manner. The current implementation uses 25 waypoints for each direction.

Each waypoint was executed sequentially using PyBullet position control. For each target waypoint, the controller repeatedly commanded the desired arm joint positions while stepping the simulator until the maximum joint error dropped below a waypoint tolerance. Once the waypoint was reached, execution advanced to the next waypoint. This same procedure was applied for both the forward and return trajectories.

This trajectory-execution layer serves as a stepping stone between static initialization and future planner-driven motion execution. Although RRT* and explicit PD tracking are not yet implemented, the current waypoint-execution pipeline already demonstrates a meaningful classical motion component for Milestone 1.

2.7 Logging and post-processing

To make the trajectory execution measurable and reportable, a logger was added to record phase labels, waypoint indices, simulation steps, target joints, actual joints, joint errors, and end-effector pose during execution. The experiment saved a trajectory log to results/m1/trajectory_log.json, which was then post-processed to generate summary metrics and visualization plots.

The post-processing pipeline computed total logged samples, phase-wise sample counts, waypoint coverage, joint tracking error statistics, and end-effector workspace ranges. It also produced trajectory figures such as end-effector position versus global sample index, end-effector XZ path, and maximum joint tracking error versus global sample index. These plots provide quantitative evidence that the execution is smooth, repeatable, and bounded.

3. Preliminary Results

The current implementation successfully establishes the simulation and classical motion-execution foundation for the project. The PyBullet environment launches correctly, the Franka Panda loads with the expected seven controllable arm joints and two gripper joints, and the scene contains a grounded table and a cube placed on the tabletop. The robot is positioned beside the table and oriented toward the task workspace, producing a realistic tabletop manipulation arrangement. In the latest run, the table AABB confirmed a grounded scene with table bottom z = 0.0 and tabletop height approximately z = 0.052 m.

The extracted robot state is consistent with the commanded home pose. The reported arm joint positions were very close to the nominal home configuration, and the measured joint velocities were near zero, showing that initialization was stable. The simulator also correctly reported end-effector position and orientation, cube pose, and table geometry, indicating that robot and environment state access are functioning correctly.

Beyond static initialization, the system now executes visible joint-space motion through interpolated waypoint trajectories. In the latest experiment, the arm executed a home-to-target trajectory and a target-to-home trajectory, each with 25 waypoints. All 25 waypoints were reached successfully in both directions, and each phase completed in 112 simulation steps. The final home-return maximum joint error was approximately 0.0327 rad, indicating good round-trip repeatability.

The saved trajectory summary provides additional quantitative support. Across both phases, 224 logged samples were recorded, with 112 samples in each phase. The global maximum absolute joint error was 0.046407 rad and the global mean maximum absolute joint error was 0.038330 rad. For the home-to-target phase, the end-effector moved over workspace ranges x ∈ [0.5325, 0.6500] m, y ∈ [-0.5930, -0.4849] m, and z ∈ [0.4853, 0.6265] m. For the target-to-home phase, the corresponding ranges were x ∈ [0.5197, 0.6423] m, y ∈ [-0.5813, -0.4787] m, and z ∈ [0.4969, 0.6395] m. These nearly symmetric workspace ranges indicate repeatable execution in both directions.

The visualization results are also consistent with stable waypoint tracking. The maximum-joint-error plot shows a repeating sawtooth pattern, which is expected for sequential waypoint execution: the error increases when a new waypoint is commanded and decreases as the controller converges toward that waypoint. The end-effector position and XZ-path plots show smooth outbound and return motion with strong overlap between phases, providing qualitative evidence of repeatability. These results demonstrate that the system now supports not only scene construction and state extraction, but also a functioning classical motion-execution baseline suitable for later planner and learning integration.

At the same time, the current implementation should still be viewed as an intermediate classical baseline rather than a complete final manipulation stack. The project structure includes scaffolding for planning and controller extensions, but configuration-space RRT*, explicit PD trajectory tracking, and full pick-and-place execution are not yet completed in this milestone.

4. Future Work

The next step is to replace the manually specified joint-space target trajectory with a planner-generated trajectory. A configuration-space RRT* planner with collision checking will be added to generate feasible arm motions automatically in the tabletop environment. The planner output will then be converted into executable waypoint or time-parameterized trajectories.

After planner integration, the current reliance on PyBullet position control will be extended with a more explicit tracking-control layer. A PD-based trajectory tracking controller will be implemented so that commanded joint trajectories can be followed with clearer control logic and gain tuning. This will provide a stronger classical baseline for later comparison against residual reinforcement learning.

Finally, the trajectory execution pipeline will be integrated into a nominal pick-and-place demonstration with the cube object, after which perturbations, residual PPO control, and robustness evaluation will be introduced in later milestones. The current Milestone 1 implementation therefore provides the essential base on which planning, control, and learning components can be layered incrementally.

2.8 Blocked-direct-path target selection

To move beyond trivial direct-path cases, a blocked-path validation harness was added. The system first evaluates a small fixed candidate bank of joint targets, and if none produce a blocked straight edge, it performs a seeded randomized search over joint-space samples. A candidate is accepted only if:

the start state is collision-free,
the goal state is collision-free, and
the direct joint-space edge from start to goal is not collision-free.

This mechanism ensures that the planner is tested on a genuinely nontrivial problem rather than a case where straight-line interpolation is sufficient. In the final seeded configuration, the blocked target is reproducibly selected at random sample 114.

2.9 RRT and RRT* planning

The planner was then extended from direct-path checking into a true sampling-based search. The first version was an RRT-style planner that:

samples random joint configurations,
finds the nearest existing tree node,
steers toward the sample by a limited step size,
rejects invalid or colliding configurations/edges,
attempts direct connection to the goal when possible, and
reconstructs a sparse path from the goal back to the root.

This baseline was then upgraded to RRT* by adding:

local neighbor search,
best-parent selection based on path cost,
rewiring of nearby nodes when a cheaper connection is found, and
tracking of the best goal node found over the full iteration budget.

This means the planner now improves path quality over time instead of simply returning the first feasible solution. The best-goal cost is printed periodically during planning, alongside the number of nodes and cumulative rewires, so that optimization behavior can be observed directly.

2.11 Replanning return motion from actual reached state

An important robustness fix was added after observing that return execution could fail if the forward motion stopped early and the return plan still assumed the ideal target state as its start. The pipeline was updated so that return planning always starts from the robot’s actual measured joint configuration after the forward segment, rather than the nominal goal configuration alone.

This change eliminated severe return-start mismatches and allowed safe recovery when forward tracking or planning did not exactly end at the intended goal. In the final seeded blocked-case runs, both directions succeed, but this change remains important for robustness and is part of the current planning/execution architecture.

2.12 Reproducibility through seeded planning

Because sampling-based planning is stochastic, a planner random seed was introduced so that blocked-target selection, tree growth, and final planner outputs are reproducible. In the seeded validation, repeated runs selected the same blocked target (sample 114), produced the same best goal costs, same sparse-path reductions, same waypoint counts, same simulation-step counts, and the same final home-return error. This reproducibility is essential for fair comparison between planner variants and for report-quality evaluation.

4. Future Work

The most important next step is to move from joint-space motion generation toward a full manipulation task pipeline. The current system already plans and executes arm motions in a tabletop scene, but the next milestone should integrate these motions into a nominal pick-and-place routine for the cube, including grasp approach, lift, transfer, and placement phases.

A second important next direction is to strengthen evaluation. The seeded planner now makes it possible to run controlled ablations comparing:

RRT,
RRT*,
RRT* with shortcut smoothing, and
possibly different controller gains or waypoint densities.

These comparisons can quantify the benefits of optimization and smoothing in terms of planner cost, sparse path length, dense waypoint count, simulation-step count, and final tracking error.

The longer-term learning objective of the project remains unchanged: use the planner/controller stack as a nominal classical backbone and then add residual learning on top of it. Once the manipulation pipeline is complete, future work will introduce disturbance injection, task-level reward design, and residual reinforcement learning to compensate for execution mismatch and improve robustness relative to purely classical planning and control. This would allow the project to study the interaction between planner-guided structure and learned adaptation in a controlled tabletop manipulation setting.

Finally, the current collision model is still conservative and the planner remains relatively expensive on hard blocked cases. Future improvements can refine collision filtering, improve planner efficiency, and introduce richer post-processing or time-parameterization of paths. However, the present implementation already provides a strong classical foundation upon which those future planning and learning extensions can be layered.
