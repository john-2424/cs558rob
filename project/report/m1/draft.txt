# Updated Milestone 1 Report Draft

## 1. Introduction

This project investigates a hybrid manipulation framework that combines classical motion planning with learning-based adaptation for robust tabletop robotic manipulation. The long-term goal, as stated in the proposal, is to use a structured planner-controller backbone and later augment it with residual reinforcement learning so that the system retains the feasibility and interpretability of classical robotics while improving robustness to execution mismatch and uncertainty. The chosen platform is the Franka Emika Panda 7-DoF manipulator simulated in PyBullet, with a tabletop cube manipulation task as the primary experimental setting. The proposal specifically defined Milestone 1 as the planning and control foundation: load the Panda in PyBullet, implement configuration-space RRT*, generate executable trajectories, implement PD tracking, and demonstrate nominal pick-and-place execution. 

The original Milestone 1 draft already established a strong simulation-and-planning baseline, including scene construction, environment state extraction, blocked-case joint-space planning, custom PD tracking, seeded RRT/RRT* planning, shortcut smoothing, and reproducible round-trip motion execution. However, that earlier draft still framed the manipulation pipeline as future work. During this implementation cycle, the system was extended from a pure motion-planning benchmark into a full nominal pick-and-place demo. The final milestone system now satisfies the professor’s requested Milestone 1 elements: simulation environment setup, robot loading, robot/environment state extraction, implementation of a classical component (planning and control), and preliminary experimental validation in a nominal manipulation task.  

The main technical contribution of the current implementation is an end-to-end classical manipulation stack for nominal tabletop pick-and-place. The final system includes: geometry-aware scene setup; a custom joint-space PD controller; collision-aware joint-space planning with RRT*; interpolated trajectory execution; inverse kinematics for Cartesian task phases; explicit gripper open/close control; contact-gated grasp validation; object attachment for stable transport after a validated grasp; attached-object-aware planning logic; and final placement/recovery phases including retreat and return-home. After several rounds of debugging and tuning, the system successfully executes the sequence: home, pre-grasp, grasp descend, grasp closure, validated pickup, lift, transfer, place descend, release, retreat, and return-home. In the final successful run, all major phases succeed, planner direct paths are found throughout the nominal demo, grasp validation passes with real contact, the final cube pose is close to the intended place pose, and the final home error is small (`max abs error ≈ 8.34e-4`).

This milestone therefore serves two roles. First, it demonstrates a complete classical manipulation baseline that is strong enough to stand alone. Second, it provides the nominal structured controller needed for later milestones on planner-guided residual RL, as originally proposed. In that sense, the final Milestone 1 result is not just a planning demo or just a control demo; it is the classical backbone for the full project.

## 2. Methods

### 2.1 Simulation environment setup

The simulation environment is built in PyBullet with a fixed-base Franka Emika Panda manipulator, a ground plane, a rectangular table, and a cube object placed on the tabletop. PyBullet was chosen because it supports URDF-based robot loading, rigid-body dynamics, collision queries, state access, and efficient prototyping of both classical control and later reinforcement learning components. The simulator also provides convenient access to object ground-truth pose, which matches the proposal’s plan to begin with nominal manipulation in simulation before adding perturbations in later milestones. 

To make the scene reproducible, geometry-aware placement was used rather than hard-coding all heights. After loading the table, the code measures its axis-aligned bounding box (AABB), translates the table so its lowest point rests on the plane, computes the tabletop height, and then places the cube so that its bottom rests on the table with a small clearance. Earlier logs from the baseline confirmed a table top height near `z ≈ 0.052 m`, and the final system retains this geometry-aware scene setup as part of the active pick-and-place pipeline. This was important because early grasping failures were strongly influenced by small geometric inconsistencies and poor robot-workspace alignment. 

### 2.2 Software organization

The project is organized into modular components so that planning, control, task sequencing, and evaluation can evolve without major restructuring. The key modules are: `sim/` for environment loading and robot state/control interfaces, `controller/` for the custom PD implementation, `planner/` for collision-aware direct-path/RRT/RRT* search, `trajectory/` for interpolation into executable waypoints, `demo/` for the task-level pick-and-place pipeline, and `main.py` as the experiment entry point. This structure was already present in the earlier draft and was validated again during the transition from blocked-path motion planning to full pick-and-place execution. 

### 2.3 Robot loading and state extraction

The Panda URDF is loaded as a fixed-base manipulator. Joint metadata is extracted from PyBullet to determine the 7 arm joints, 2 gripper joints, and the end-effector/grasp target link. The simulator is queried for joint positions, joint velocities, end-effector pose, cube pose, and table AABB geometry. Earlier validated startup logs showed the expected Panda indices (`[0,1,2,3,4,5,6]` for arm joints and `[9,10]` for gripper joints) and near-zero startup velocities. This same state-access pipeline is reused by the final demo for planning, execution, and grasp verification. 

### 2.4 Stable home configuration and geometry tuning

A nominal home joint configuration is used as the default start state for all experiments. After the scene is placed, the robot is reset to home and commanded to hold that configuration so that all runs begin from a repeatable initialization. During manipulation development, it became clear that robot base placement relative to the table was critical. When the Panda base was too far from the table, the arm often approached the cube in a stretched posture and the grasp target required ad hoc Y-bias compensation. After moving the robot base closer to the workspace, the earlier manual Y correction became unnecessary, and the grasp alignment improved substantially. The final working setup uses the closer base placement and `GRASP_TARGET_BIAS_Y = 0.0`, which proved more physically meaningful and more stable than compensating for poor workspace geometry with a manual target shift. This iterative geometry correction was a major practical lesson from the milestone implementation.

### 2.5 Custom joint-space PD control

A key improvement beyond the earliest baseline is the use of a custom joint-space PD controller rather than relying only on PyBullet’s built-in position servoing. The controller computes desired joint velocity commands from position and velocity error:
[
\dot{q}*{cmd} = K_p (q*{des} - q) + K_d (\dot{q}_{des} - \dot{q}),
]
and applies these through PyBullet joint velocity control. This separates path generation from low-level tracking and gives direct access to control gains and commanded velocities. The execution loop records per-waypoint tracking error and controller effort (`last_max_vel_cmd`), allowing convergence behavior to be observed directly. This PD tracking architecture was already part of the earlier planning baseline and remains central in the final pick-and-place system. 

### 2.6 Waypoint execution, tolerance logic, and logging

Motion execution is performed waypoint-by-waypoint. For each desired joint target, the controller commands the arm (and, when needed, the gripper), the simulator steps forward, the current robot state is queried, and the maximum absolute joint error is monitored. The system advances only when a waypoint is reached within tolerance or when it exhausts the configured settle budget. This execution loop is used for both planner-generated motion and task-phase motion in the final pick-and-place pipeline.

The logger records the phase label, waypoint index, simulation step, target joints, actual joints, joint error, end-effector position, and orientation, saving the results to `results/m1/trajectory_log.json`. This logging mechanism was important both for the earlier blocked-case planning experiments and for diagnosing manipulation issues such as overshoot, undershoot, and grasp misalignment. 

### 2.7 Collision-aware planning: direct path, RRT, and RRT*

The planning stack began with a direct-path validator in joint space and was extended to RRT and then RRT*. The planner checks joint limits, collision-free validity of start/goal states, and collision-free validity of interpolated edges. The RRT* version adds neighbor search, best-parent selection, rewiring, and cost tracking so that path quality can improve over the planning budget. The earlier milestone draft documented this progression in detail, including seeded blocked-path target selection, shortcut smoothing, and reproducible blocked-case evaluation. That earlier work still stands as the planner foundation for the present milestone. 

### 2.8 Transition from blocked-path benchmark to nominal pick-and-place

The most important extension made during this chat was the transition from a joint-space planning benchmark into a full task-level manipulation routine. A new task pipeline was implemented with the following phases:

1. home
2. pre-grasp
3. grasp descend
4. close gripper
5. validate grasp and attach object
6. lift
7. transfer to pre-place
8. place descend
9. detach/release
10. open gripper
11. retreat
12. return-home

Each Cartesian task phase is converted into a joint target via inverse kinematics and then executed through the same planner/execution stack used in the earlier motion-planning work. This allowed the project to preserve the modularity of the classical backbone while extending it to a nominal manipulation task, exactly as envisioned in the proposal’s Milestone 1 description.

### 2.9 Grasping pipeline and manipulation debugging

Implementing a nominal pick-and-place routine required several iterations of debugging and refinement.

The first issue was that the cube could appear to move even when the robot had not clearly reached a physical grasp. This occurred because the initial implementation attached the object after gripper closure without verifying that the hand was actually close enough or in contact with the cube. To fix this, a contact-gated grasp validation step was added. The system now checks geometric proximity and contact information before attaching the object, and only after the grasp is validated does it create the fixed constraint used to stabilize transport. This removed the earlier “fake grasp” behavior and made failures honest.

The second issue was target-frame alignment. Early in development, the robot could move down properly in z while still being visibly in front of or behind the cube in y. This led to several experiments with end-effector frame interpretation, finger-position debugging, pinch-point reasoning, and manual target biasing. Ultimately, the most effective fix was not a permanent end-effector Y bias, but rather moving the Panda base closer to the table so that the grasp occurred in a more comfortable reachable posture. With the closer base, the previously needed manual Y correction became unnecessary and was set back to zero.

The third issue was manipulation planning in contact-rich phases. During lift, transport, and placement, the planner could incorrectly declare the start state in collision, causing fallback to coarse interpolated motion and producing visibly jerky behavior. This was traced to the RRT* collision checker being too conservative for attached-object phases. The planner was then updated to allow appropriate attached-object contacts in both configuration checks and edge checks. Once that fix was made, the nominal demo progressed much more cleanly and with fewer jerky fallback motions. This was a substantial improvement over the earlier contact-blind planning behavior.

### 2.10 Placement and return-home refinement

Placement also required refinement. Earlier versions used snap-to-place logic before release, which made the cube appear to arrive at the target even if the robot did not physically place it there precisely. For the final version, the place target was moved to a more reachable region toward the robot, reducing stretch and making the placement more naturally reachable. The successful nominal run places the cube close to the intended place target and then retreats successfully. A final settle-at-home phase was also added after return-home so that the home configuration converges more cleanly before reporting final error. In the final successful run, the cube ended at approximately `[0.5198, 0.2203, 0.0760]` and the final home return maximum absolute joint error was approximately `8.26e-4`, which is substantially cleaner than earlier runs that ended much closer to the tolerance boundary.

## 3. Preliminary Experimental Results

The final milestone implementation now demonstrates a complete nominal classical pick-and-place routine rather than only a planning benchmark. This is an important update relative to the earlier draft, which still framed pick-and-place as future work. In the final successful run, all key phases execute successfully: pre-grasp, grasp descend, gripper closure, validated object attachment after retry descend, lift, transfer, placement, retreat, and return-home. The final cube pose is near the intended place target and the final home error is small, showing both successful manipulation and successful reset. Specifically, one successful run reports the final cube pose near `[0.5198, 0.2203, 0.0760]` and a final home maximum absolute error of about `8.26e-4`; another closely related successful run reports a similar result with final home error about `8.34e-4`.

A major positive result is that the grasp is no longer accepted artificially. In an earlier intermediate stage, the system could attach the cube even when the robot had not physically reached a convincing grasp, which was traced to unconditional object attachment after gripper closure. The final validated version no longer does this. Instead, grasp acceptance depends on contact and geometry checks. In the successful run, the retry grasp validation reports `ee_to_cube_dist = 0.00831`, `min_finger_to_cube_dist = 0.02786`, `total_contacts = 5`, `finger_contacts = 5`, and `ready = True`, after which the object is attached and transported. This is strong evidence that the final nominal pick-and-place is contact-validated rather than purely scripted. 

The final system also preserves the value of the earlier planning work. The original blocked-case planning benchmark established that the planner was capable of handling nontrivial joint-space problems, not only easy direct paths. That earlier work implemented seeded blocked direct-path selection, RRT/RRT* search, shortcut smoothing, and reproducible evaluation. Although the nominal pick-and-place run often uses direct planner connections because the task is easier than the blocked benchmark, the manipulation pipeline is built on the same planning/control foundation and therefore inherits the same modular architecture and reproducible setup philosophy. 

From a qualitative standpoint, the final demo also taught several practical lessons that are relevant to classical manipulation:

* workspace geometry matters as much as controller tuning;
* manual target biases can temporarily compensate for poor geometry, but are often removed once the robot base is placed properly;
* grasp validation must be explicit, otherwise the simulator may give misleading success signals;
* attached-object motion requires collision filtering that respects grasped-object context;
* and reachable place targets produce substantially more stable and natural placement behavior.

These lessons were not merely conceptual; they emerged directly from iterative debugging of the actual final implementation.

Overall, the current Milestone 1 result demonstrates all of the professor’s requested components:

* a simulation environment in PyBullet,
* robot loading,
* robot and environment state extraction,
* implementation of a classical component through planning and control,
* and preliminary results through a successful nominal pick-and-place demonstration.

## 4. Future Work

The long-term objective remains the one defined in the original proposal: planner-guided residual reinforcement learning for robust robotic manipulation. With the nominal classical pick-and-place now working, the next milestone can shift from establishing the backbone to improving robustness under uncertainty. The proposal explicitly identifies controlled pose perturbations, residual PPO, and comparison against planner-only and RL-only baselines as future milestones, and the current classical stack is now mature enough to support that transition. 

The first immediate next step is to introduce controlled perturbations in object pose and/or execution so that the current nominal pipeline can be stress-tested. This will help quantify where the classical planner-controller stack begins to fail and where learned residual corrections may provide value. The second next step is to integrate a bounded residual RL policy on top of the current PD/planner backbone, following the proposal’s intended architecture:
[
\tau_{total} = \tau_{PD} + \tau_{RL}.
]
Although the current simulator implementation uses velocity-based PD actuation inside PyBullet rather than direct real-world torque control, the conceptual structure of a nominal classical controller plus a bounded learned residual remains the same. 

A second future-work direction is stronger evaluation. The earlier draft already proposed ablations comparing RRT, RRT*, shortcut smoothing, and controller settings, and those experiments are still valuable. With the nominal pick-and-place now complete, the evaluation can be expanded to include:

* pick-and-place success rate under repeated nominal trials,
* grasp success before and after retry descend,
* placement error relative to the target place pose,
* sensitivity to object pose perturbation,
* planner-only versus planner-plus-residual comparisons in disturbed settings.

Finally, richer task extensions from the proposal remain open for later milestones, including drawer opening, SAC comparison, and peg-in-hole evaluation if time permits. However, the guaranteed primary path remains planner-guided residual PPO for pick-and-place with quantitative robustness evaluation. The current milestone result provides the exact classical nominal baseline needed to support that roadmap.
